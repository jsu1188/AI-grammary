# CheckWord 인수인계서

- 작성일: 2026-06-07
- 작업 경로: `checkword`
- 목적: 현재 프로젝트 구조, 저장 흐름, 최근 반영 사항, 남아 있는 이슈를 빠르게 파악할 수 있도록 최신 상태를 정리

## 1. 프로젝트 개요

CheckWord는 사용자가 입력하거나 브라우저/클립보드에서 읽어온 문장을 기반으로 다음 기능을 제공하는 문장 보조 도구입니다.

- 맞춤법 교정
- 요약
- 문체/말투 변환
- 제목 추천
- 점수/평가
- 기록 조회 및 재호출

구성은 크게 아래 3개입니다.

- `client/`: 데스크톱 UI
- `server/`: FastAPI 서버 및 DB 연동
- `browser_extension/`: 브라우저 실시간 읽기 확장 기능

## 2. 주요 디렉터리

- `checkword/client/ui/main_window.py`
  - 메인 창 구성
  - 입력 이벤트 수신
  - 실시간 모드 상태 관리
- `checkword/client/ui/result_panel.py`
  - 탭별 결과 패널
  - 기록 목록 / 기록 상세 / 미호출 기능 버튼
- `checkword/client/core/auth_api_client.py`
  - 서버 API 호출 래퍼
- `checkword/client/input/ai_grammary_text_reader.py`
  - 클립보드/입력 감지 관련 로직
- `checkword/browser_extension/content.js`
  - 브라우저 DOM에서 텍스트 읽기
  - 활성 입력창 추적
- `checkword/server/main.py`
  - FastAPI 엔드포인트
  - 기록 저장/조회/삭제

## 3. DB 저장 구조

현재 저장 구조는 “원문 요청 1건 + 기능별 결과 분리 저장” 기준입니다.

### 요청 원문

- `analysis_requests`
  - 사용자가 실제로 요청한 원문 저장
  - 기록 목록의 기준이 되는 테이블

### 기능별 결과

- `spelling_results`
  - 맞춤법 결과 저장
- `summary_results`
  - 요약 결과 저장
- `tone_results`
  - 문체/말투 결과 저장
- `evaluation_results`
  - 평가/점수 결과 저장
- `title_results`
  - 제목 추천 결과 저장

즉, 원문은 `analysis_requests`에 저장되고, 각 기능 결과는 별도 테이블에 매핑됩니다.

## 4. 현재 정상 동작 기준 흐름

### 4-1. 최초 요청

1. 원문 요청 발생
2. `analysis_requests`에 원문 저장
3. 호출한 기능 결과만 해당 결과 테이블에 저장

예시:

- 맞춤법 호출 시
  - 원문: `analysis_requests`
  - 교정 결과: `spelling_results`

- 요약 호출 시
  - 원문: `analysis_requests`
  - 요약 결과: `summary_results`

### 4-2. 기록 조회

- 기록 목록은 `analysis_requests` 기준으로 조회
- 상세 진입 시 해당 원문에 연결된 결과들을 각 테이블에서 조회
- 아직 호출하지 않은 기능은 “미호출 기능”으로 표시

### 4-3. 기록에서 추가 호출

- 과거 원문을 기록에서 다시 선택 가능
- 이전에 호출하지 않았던 기능만 상세 화면에서 추가 호출 가능
- 추가 호출 후 결과는 해당 결과 테이블에 저장

## 5. 최근 반영된 사항

### 5-1. 기록 목록 기준 정리

- 기록 목록은 원문 요청(`analysis_requests`) 기준으로 동작
- 상세에서는 해당 원문으로부터 파생된 결과들을 묶어서 보여주는 구조로 정리됨

### 5-2. 맞춤법/요약/문체/제목/평가 분리 저장

- 교정문을 원문 요청 테이블에 같이 넣지 않고 결과 테이블에 분리 저장하는 구조로 정리됨
- 기능별 결과 누락 여부를 상세 화면에서 확인 가능

### 5-3. 기록 상세의 미호출 기능 버튼

- 기록 상세에 `미호출 기능` 버튼 영역 존재
- 아직 호출하지 않은 기능만 버튼으로 노출됨
- 버튼 클릭 시 해당 원문 기준으로 추가 API 호출 가능

### 5-4. 문체/말투 추가 호출 입력 방식

- 기록 상세에서 문체/말투가 미호출 상태일 경우
- 메인 탭 입력칸이 아니라 상세 영역 내부에서 별도 입력을 받아 호출하도록 처리됨
- 즉, 기록에서 과거 원문을 선택한 뒤 원하는 문체/말투 지시어를 넣어 추가 호출 가능

### 5-5. 기록 삭제 기능 추가

- 서버 엔드포인트 추가:
  - `DELETE /history/requests/{request_id}`
- 클라이언트 API 추가:
  - `AuthAPIClient.delete_history_request`
- 기록 목록 항목 우측에 삭제 버튼 추가
- 삭제 시 해당 요청 원문과 연결 결과 정리용 흐름이 들어가 있음

### 5-6. 기록 시간 표시

- 기록 화면 시간 표시는 로컬 기준으로 보이도록 포맷 처리 로직이 들어가 있음

### 5-7. 메인 탭 우하단 목록 버튼 정리

- `텍스트`, `맞춤법`, `요약`, `문체/말투` 탭의 우하단 목록 버튼은 제거된 상태
- 기록 관련 접근은 `기록` 탭 중심 구조로 정리됨

## 6. 현재 남아 있는 주요 이슈

### 6-1. 기록 상세 패널 UI 불안정

가장 주의해야 할 부분입니다.

- `client/ui/result_panel.py`의 기록 상세 레이아웃이 여러 차례 수정되면서 구조가 많이 흔들린 상태
- 특히 아래 증상이 반복적으로 발생했음
  - 상세 하단에 불필요한 빈 공간이 크게 남는 문제
  - 필드 높이/스크롤 처리 꼬임
  - 텍스트가 박스에 비해 작게 잡히거나 겹쳐 보이는 문제
  - 일부 상태에서 상세 내용이 거의 없는 것처럼 보이는 문제

즉, 기록 상세 패널은 현재 “기능은 살아 있으나 UI는 매우 민감한 상태”로 보는 것이 맞습니다.

### 6-2. `self.history_panel` 잘못 덮어쓰기 가능성

`result_panel.py`에서 기록 패널 참조가 다른 페이지 생성 과정에서 다시 할당되는 흔적이 있습니다.

- `_create_auth_page`
- `_create_prompt_page`

위 구간에서 `self.history_panel = panel` 같은 재할당이 들어가 있으면 기록 패널 참조가 오염될 수 있습니다.

이 부분은 기록 상세 UI가 비정상 크기로 보이는 원인 후보 중 하나입니다.

### 6-3. 브라우저 실시간 읽기 모드의 과호출 가능성

아직 해결되지 않은 이슈입니다.

원인:

- `browser_extension/content.js`
  - `activeEditable()`이 정확한 활성 입력창을 못 찾으면 fallback 로직으로 내려감
  - `fallbackEditable()` / `bestEditableCandidate()`가 페이지 내 다른 `textarea`, `input`, `contenteditable`을 후보로 잡을 수 있음
- `main_window.py`
  - 브라우저 입력 이벤트가 들어오면 `last_input`을 갱신하고 자동 맞춤법 호출 예약을 수행함

결과:

- 사용자가 교정하려는 텍스트가 아닌 다른 DOM 텍스트도 읽혀서 API가 불필요하게 호출될 수 있음

즉, 실시간 브라우저 모드는 현재 API 낭비 가능성이 있습니다.

### 6-4. 서버 라우트 변경 후 재시작 필요

- 기록 삭제 등 API를 수정한 뒤에는 기존 실행 서버가 예전 코드로 떠 있으면 `Not Found`가 발생할 수 있음
- 이 경우 서버 프로세스를 완전히 재시작해야 반영됩니다

## 7. 현재 확인된 핵심 API

- `GET /history/requests`
  - 기록 목록 조회
- `DELETE /history/requests/{request_id}`
  - 기록 삭제

그 외 맞춤법/요약/문체/제목/평가 호출 API는 기존 기능대로 사용

## 8. 현재 작업 시 주의사항

1. 기록 상세 UI는 작은 레이아웃 수정도 연쇄적으로 깨질 수 있음
2. 기록 목록/상세 로직 수정 시 `analysis_requests` 기준 구조를 유지해야 함
3. 교정문은 원문 요청 테이블에 넣지 말고 `spelling_results`에 저장해야 함
4. 요약/문체/평가/제목도 각각 결과 테이블에 저장되어야 함
5. 브라우저 실시간 입력은 활성 DOM 판별을 더 엄격하게 하지 않으면 API 과호출이 계속 남음
