Checkword 디버그 기록
작성일: 2026-05-28

1. 원본 수정 버튼이 보이지 않던 문제
- 증상: "실시간 인식 사용"과 "수정 방식 사용"을 체크했는데도 "원본 수정" 버튼이 보이지 않았다.
- 원인: 버튼 표시 조건이 "선택 영역 인식 사용" 모드에만 묶여 있었다.
- 수정:
  - 입력 모드가 "선택 영역"이면 버튼을 표시하도록 유지했다.
  - 입력 모드가 "실시간"이고 "수정 방식 사용"이 켜져 있어도 버튼을 표시하도록 수정했다.
  - "수정 방식 사용" 체크박스 변경 시에도 버튼 표시 상태가 즉시 갱신되도록 연결했다.
  - 버튼 표시 판단은 result_panel.py가 아니라 main_window.py에서만 담당하도록 정리했다.

2. 계속 테스트 모드 교정 결과가 나오던 문제
- 증상: OpenAI 코드를 적용했는데도 결과에 "(교정 완료 - 테스트 모드)"가 계속 표시됐다.
- 원인: 8765 포트에 예전 FastAPI 서버가 계속 떠 있었고, 클라이언트가 그 서버를 정상 서버로 재사용했다.
- 수정:
  - 서버 API 버전 정보를 추가했다.
  - /server-info 엔드포인트를 추가했다.
  - LocalServer가 api_version="openai-separated-v1"인 서버만 정상 서버로 인정하도록 수정했다.
  - 이전 버전 서버가 8765 포트를 사용 중이면 기존 서버를 종료하라는 오류를 띄우도록 했다.

3. OPENAI_API_KEY를 인식하지 못하던 문제
- 증상: "OPENAI_API_KEY 환경변수가 설정되어 있지 않습니다." 오류가 발생했다.
- 원인:
  - checkword/server/.env 파일에 OPENAI_API_KEY 항목이 없었다.
  - load_dotenv()가 실행 위치에 따라 .env를 못 찾을 수 있었다.
- 수정:
  - checkword/server/.env에 OPENAI_API_KEY와 OPENAI_MODEL 항목을 추가했다.
  - .env가 실수로 커밋되지 않도록 checkword/server/.gitignore를 추가했다.
  - ai_service.py가 실행 위치와 상관없이 checkword/server/.env를 절대 경로로 읽도록 수정했다.
  - 서버 터미널에 API 키를 마스킹한 설정 로그가 찍히도록 했다.

4. OpenAI API 호출이 너무 자주 발생해 프로그램이 멈추던 문제
- 증상: 라이브 리딩 중 텍스트가 조금만 바뀌어도 OpenAI 요청이 계속 발생해 프로그램이 버벅였다.
- 원인: 텍스트 변경 이벤트마다 즉시 run_spell_check()가 실행됐다.
- 수정:
  - 자동 맞춤법 검사는 2초 디바운스를 거친 뒤 실행되도록 했다.
  - OpenAI 요청은 UI 스레드가 아니라 백그라운드 스레드에서 실행하도록 바꿨다.
  - OpenAI 요청이 진행 중일 때는 추가 API 호출이 발생하지 않도록 막았다.
  - 요청 중 텍스트가 바뀌면 최신 텍스트를 기억해두고, 현재 요청이 끝난 뒤 다시 2초 대기 후 한 번만 호출하도록 했다.

5. OpenAI API 호출 중에도 라이브 리딩이 계속 돌던 문제
- 증상: API 중복 호출은 막혔지만, HWP/Word 라이브 리딩은 OpenAI 요청 중에도 계속 실행됐다.
- 원인: spell_check_in_progress는 API 호출만 막고, 실시간 모니터의 poll_snapshot() 호출은 막지 않았다.
- 수정:
  - realtime_text_monitor.py에 set_polling_paused()를 추가했다.
  - OpenAI 맞춤법 요청이 시작되면 라이브 리딩 자체를 일시정지하도록 했다.
  - OpenAI 요청이 성공하거나 실패하면 라이브 리딩을 다시 켜도록 했다.
  - reset_session_state()에서도 라이브 리딩 일시정지 상태를 해제하도록 했다.

6. HWP 라이브 리딩이 한글 입력과 클립보드에 간섭하던 문제
- 증상:
  - 한글 입력 중 "안" 같은 글자가 "아ㄴ"처럼 깨질 수 있었다.
  - 클립보드에 오늘 날짜 같은 예상하지 못한 값이 복사되는 일이 있었다.
- 원인:
  - 실시간 모니터가 매 루프마다 전체 실시간 텍스트와 선택영역 텍스트를 둘 다 폴링했다.
  - HWP 선택영역 리딩은 Ctrl+C 기반 클립보드 경로를 사용했기 때문에, 실시간 모드에서도 클립보드가 건드려질 수 있었다.
  - 임시로 추가했던 HWP GetTextFile 후보도 입력 중 간섭 가능성이 있었다.
- 수정:
  - 실시간 모드에서는 전체 문서 라이브 텍스트만 폴링하도록 했다.
  - 선택 영역 모드에서는 선택된 텍스트만 폴링하도록 했다.
  - 실시간 모드에서는 선택영역 폴링이 실행되지 않게 막았다.
  - HWP도 키 입력 중 0.55초 동안 폴링을 쉬도록 추가했다.
  - 실시간 HWP 리딩에서 GetTextFile 사용을 제거했다.
  - 문서를 건드리지 않는 Unicode NFC 정규화만 유지했다.

7. OpenAI 기능별 API 분리
- 변경:
  - 맞춤법 교정은 /correct 엔드포인트를 사용하도록 했다.
  - 글 요약은 /summary 엔드포인트를 사용하도록 했다.
  - client/core/ai_client.py에 correct_spelling()과 summarize()를 분리했다.
  - server/ai_service.py에 correct_spelling()과 summarize_text()를 분리했다.

8. 서버 터미널 OpenAI 요청 로그 추가
- 변경:
  - OpenAI 요청 시작, 성공, 실패 로그를 서버 터미널에 출력하도록 했다.
  - 로그에는 기능 이름, 입력 글자 수, 줄 수, 모델명, 소요 시간, 출력 글자 수가 표시된다.
  - /correct 또는 /summary가 실제 호출되는지 서버 터미널에서 확인할 수 있게 했다.

9. "젝아 문뻡 직히귀뽝쐐옄" 리딩/표시 문제 조사
- 증상:
  - HWP 화면에는 "젲이 문뻡 직히귀뽝쐐옄"처럼 보이는 문장이 있었다.
  - 앱 오른쪽 텍스트 영역에서는 일부 글자가 빠지거나 다르게 보였다.
  - 사용자는 브라우저, 텍스트파일, 클립보드 방식에서도 비슷하게 깨져 보인다고 보고했다.
- 조사 결과:
  - ui_input_events.log에는 실제 리더가 "젲이 문뻡 직히귀뽝쐐옄" 문자열을 받은 기록이 있었다.
  - 즉 일부 경우에는 리딩 값 자체는 들어왔지만, UI 표시 단계에서 특이한 한글 음절이 빠져 보였다.
  - 원인은 앱 전역 폰트로 사용하던 A2Z-Medium.ttf가 모든 한글 완성형 음절을 제대로 표시하지 못한 것으로 판단했다.
- 수정:
  - main_window.py에서 QApplication 전역 폰트를 A2Z-Medium 대신 "Malgun Gothic"으로 변경했다.
  - result_panel.py의 QTextEdit에도 "Malgun Gothic"을 직접 지정했다.
  - QTextEdit 내부 QTextDocument 기본 폰트도 "Malgun Gothic"으로 강제했다.
  - 원문/맞춤법/요약/문체 결과를 setPlainText로 넣기 전에 문서 폰트를 다시 적용하도록 했다.

10. HWP 라이브 리딩 후보 경로 변경
- 증상:
  - HWP COM InitScan/GetText 경로가 "안녕하세요"를 "앙녕항셍용"처럼 잘못 반환하는 사례가 있었다.
  - "제가"가 "젲이"처럼 읽히거나, 자모 조합이 섞이는 사례도 있었다.
- 조사 결과:
  - InitScan/GetText, GetTextFile(TEXT), HWPML2X, UIA 후보를 번갈아 확인했다.
  - HWP가 제공하는 COM 텍스트 계열 API가 문서 내부의 오타/비표준 입력을 그대로 또는 이상하게 변환해 반환하는 경우가 있었다.
  - UIA는 화면에 보이는 개별 줄을 후보로 반환하지만, 전체 문서가 아니라 부분 텍스트만 반환하는 경우가 많았다.
- 수정:
  - HWP 전체 라이브 리딩에서 COM 결과와 UIA 결과를 둘 다 확인하도록 했다.
  - UIA 결과가 충분히 긴 경우 UIA를 우선 사용하고, 너무 짧은 부분 텍스트이면 COM 결과를 유지하도록 했다.
  - HWPML2X XML의 BODY/P/TEXT/CHAR를 파싱하는 읽기 경로를 추가했다.
  - 선택영역 인식에서는 Ctrl+C 클립보드보다 HWPML2X selection을 먼저 시도하도록 했다.
  - HWPML2X selection 실패 시에만 클립보드 fallback을 사용하도록 했다.

11. "젝아 문뻡 직히귀뽝쐐옄" 교정 프롬프트 조정
- 증상:
  - OpenAI가 "젲이 문뻡 직히귀뽝쐐옄"을 "제가 문법 지키기 챌린지예요"처럼 원문에 없는 의미로 복원했다.
  - 이전에는 너무 보수적으로 설정되어 제대로 복원하지 못했고, 이후에는 너무 자유롭게 해석해 원문에 없는 단어를 만들었다.
- 원인:
  - 이 작업은 단순 맞춤법 교정이 아니라 발음/오타 기반 의도 복원에 가깝다.
  - 프롬프트가 너무 보수적이면 복원을 못 하고, 너무 짧거나 자유로우면 창작이 섞인다.
- 수정:
  - server/ai_service.py의 맞춤법 요청 프롬프트를 "한국어 문장 복원 교정 엔진"으로 변경했다.
  - 발음 유사, 자판 실수, 된소리/거센소리, 자모 분리 오류를 고려하도록 했다.
  - 토큰 비용을 줄이기 위해 긴 규칙 대신 대표 예시 1개를 남겼다.
  - 현재 대표 예시는 다음과 같다:
    "젝아 문뻡 직히귀뽝쐐옄 -> 제가 문법 지키기 빡세요"
- 남은 문제:
  - 모델이 "챌린지예요"처럼 원문에 없는 의미를 만들 수 있다.
  - 다음 조정에서는 "원문에 없는 새 의미를 추가하지 말라", "창작하지 말라" 같은 제약과 낮은 변동성 설정을 추가하는 것이 필요하다.
